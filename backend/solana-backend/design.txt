我们为solana_staking协议构建一个后端服务，实现：
1. 查询program的global states
2. 根据传入的用户地址，查询用户当前stake相关状态
3. 查询用户的操作历史

## 1. Bounded Contexts (限界上下文)

### 1.1 Query Domain (查询领域)
负责提供查询接口，实现三个核心功能：
- 查询program的global states
- 根据用户地址查询用户当前stake相关状态
- 查询用户的操作历史

### 1.2 Event Synchronization Domain (事件同步领域)
负责从Solana链上同步和索引链上事件到数据库，为查询功能提供数据支持。

## 2. Core Domain Entities (核心领域实体)

### 2.1 GlobalState (全局状态)
- **定义**: 质押程序的全局配置和统计信息
- **属性**:
  - admin: 管理员地址
  - stakingMint: 质押代币类型
  - rewardMint: 奖励代币类型
  - stakingVault: 质押代币金库地址
  - rewardVault: 奖励代币金库地址
  - rewardRate: 奖励率 (基点，100 = 1%)
  - totalStaked: 总质押量
- **查询来源**: 从Solana链上实时查询GlobalState账户

### 2.2 UserStakeStatus (用户质押状态)
- **定义**: 用户在质押程序中的当前质押状态
- **属性**:
  - userAddress: 用户地址
  - amount: 当前质押数量
  - stakeTimestamp: 首次质押时间
  - lastClaimTime: 上次领取奖励时间
  - rewardDebt: 已领取的奖励债务
  - pendingRewards: 待领取奖励数量 (计算得出)
- **查询来源**: 从Solana链上实时查询UserStakeInfo账户

### 2.3 UserActivity (用户活动记录)
- **定义**: 用户在质押协议中的所有操作历史记录
- **属性**:
  - userAddress: 用户地址
  - programId: 程序ID
  - eventType: 事件类型 (Staked/Unstaked/RewardsClaimed)
  - rawData: 原始事件数据
  - positionDelta: 质押数量的变化量（正数表示增加，负数表示减少）
  - rewards: 操作相关的奖励数量
  - blockNumber: 区块号 (Solana中的slot)
  - txHash: 交易哈希
  - timestamp: 时间戳
- **查询来源**: 从数据库UserActivity表查询
- **说明**: positionDelta用于显示操作前后质押数量的变化，Staked事件为正数，Unstaked事件为负数

### 2.4 SyncStatus (同步状态)
- **定义**: 事件同步服务维护的同步进度状态
- **属性**:
  - programId: 程序ID
  - lastSyncBlock: 最后同步的区块号
  - initializeBlock: 程序初始化的区块号
- **职责**: 跟踪事件同步进度，确保数据完整性

## 3. Value Objects (值对象)

### 3.1 Slot (区块槽位)
- **定义**: Solana区块链中的区块标识符
- **属性**: number

### 3.2 TransactionHash (交易哈希)
- **定义**: Solana交易的唯一标识符
- **属性**: string (Base58编码)

### 3.3 Address (地址)
- **定义**: Solana账户地址 (Pubkey)
- **属性**: string (Base58编码的Pubkey)

### 3.4 TokenAmount (代币数量)
- **定义**: 代币数量，以最小单位表示
- **属性**: number (u64)

## 4. Domain Events (领域事件)
用于事件同步，从链上捕获并存储到UserActivity表

### 4.1 StakedEvent (质押事件)
- **属性**: user, amount, timestamp

### 4.2 UnstakedEvent (解质押事件)
- **属性**: user, amount, rewards, timestamp

### 4.3 RewardsClaimedEvent (奖励领取事件)
- **属性**: user, amount, timestamp

### 4.4 InitializedEvent (初始化事件)
- **属性**: authority, stakingMint, rewardMint, rewardRate, timestamp

## 5. Domain Services (领域服务)

### 5.1 RewardCalculationService (奖励计算服务)
- **职责**: 计算用户待领取的奖励数量
- **输入**: UserStakeStatus, GlobalState, currentTimestamp
- **输出**: 待领取奖励数量
- **公式**: (当前时间 - lastClaimTime) * amount * rewardRate / (10000 * 86400)

### 5.2 EventSynchronizationService (事件同步服务)
- **职责**: 从Solana链上同步事件到数据库
- **依赖**: 
  - SolanaEventFetcher (event-fetch模块): 负责从链上获取事件
  - TransactionEventsParser (event-fetch模块): 负责解析交易中的事件
  - UserActivityRepository: 持久化用户活动记录
  - SyncStatusRepository: 维护同步状态
- **输入**: programId, startSlot, endSlot, eventParserType ("permissionless" | "admin")
- **输出**: 同步的事件列表 (BaseEvent[])
- **实现流程**:
  1. 创建或获取SolanaEventFetcher实例
  2. 根据eventParserType创建对应的TransactionEventsParser
  3. 调用SolanaEventFetcher.fetchEvents()获取链上事件
  4. 返回FetchingResult.events
- **规则**: 按slot顺序同步，维护SyncStatus状态

## 6. Repositories (仓储)

### 6.1 UserActivityRepository
- **职责**: 查询和保存用户活动记录
- **方法**:
  - `save(activity: UserActivity): Promise<void>`: 保存用户活动记录
  - `findByUser(userAddress: string, programId: string): Promise<UserActivity[]>`: 查询指定用户的所有活动
  - `findByUserAndEventType(userAddress: string, programId: string, eventType: string): Promise<UserActivity[]>`: 按事件类型过滤查询

### 6.2 SyncStatusRepository
- **职责**: 查询和维护同步状态
- **方法**:
  - `findByProgram(programId: string): Promise<SyncStatus | null>`: 获取当前同步状态
  - `save(syncStatus: SyncStatus): Promise<void>`: 保存同步状态
  - `updateLastSyncBlock(programId: string, lastSyncBlock: number): Promise<void>`: 更新同步进度

### 6.3 GlobalStateRepository (链上查询)
- **职责**: 从Solana链上查询全局状态
- **方法**:
  - `getGlobalState(programId: string, stakingMint: string, stateSeed: string): Promise<GlobalState>`
    - **PDA推导**: 使用stateSeed（默认: "state"）和stakingMint推导GlobalState的PDA地址
    - **Seeds**: `[stateSeed, stakingMint]` 
    - **推导公式**: `PublicKey.findProgramAddress([stateSeed, stakingMint], programId)`
    - 使用推导出的PDA地址查询GlobalState账户

### 6.4 UserStakePositionRepository (链上查询)
- **职责**: 从Solana链上查询用户质押状态
- **方法**:
  - `getUserStakePosition(userAddress: string, programId: string, stakingMint: string, stateSeed: string, stakeSeed: string): Promise<UserStakeStatus | null>`
    - **PDA推导步骤**:
      1. 先推导GlobalState的PDA: `stateAddress = findProgramAddress([stateSeed, stakingMint], programId)`
      2. 再推导UserStakeInfo的PDA: `findProgramAddress([stakeSeed, stateAddress, userAddress], programId)`
    - **Seeds**: `[stakeSeed, stateAddress, userAddress]` 
      - stakeSeed: 默认 "stake"
      - stateAddress: 从步骤1获得
      - userAddress: 用户地址
    - 使用推导出的PDA地址查询UserStakeInfo账户

## 7. Application Services (应用服务)

### 7.1 StakingQueryService (质押查询服务)
- **职责**: 提供三个核心查询接口，协调Repository和Domain Service
- **依赖**: GlobalStateRepository, UserStakePositionRepository, UserActivityRepository, RewardCalculationService
- **方法**:
  - `getGlobalState(programId: string, stakingMint: string, stateSeed: string): Promise<GlobalState>`
    - 内部调用: GlobalStateRepository.getGlobalState(programId, stakingMint, stateSeed)
    - stateSeed默认值为"state"
  
  - `getUserStakeStatus(userAddress: string, programId: string, stakingMint: string, stateSeed: string, stakeSeed: string): Promise<UserStakeStatus>`
    - 内部调用: UserStakePositionRepository.getUserStakePosition() + RewardCalculationService计算待领取奖励
    - stateSeed默认值为"state"，stakeSeed默认值为"stake"
  
  - `getUserActivityHistory(userAddress: string, programId: string): Promise<UserActivity[]>`
    - 内部调用: UserActivityRepository.findByUser(userAddress, programId)
    - 返回结果按timestamp倒序排序

### 7.2 EventSyncService (事件同步服务)
- **职责**: 协调事件同步流程
- **依赖**: EventSynchronizationService, UserActivityRepository, SyncStatusRepository
- **方法**:
  - `syncEvents(programId: string, startSlot?: number, endSlot?: number): Promise<SyncResult>`
    - 调用EventSynchronizationService获取链上事件
    - 将BaseEvent转换为UserActivity实体
    - 通过UserActivityRepository保存到数据库
    - 更新SyncStatus
  
  - `getSyncStatus(programId: string): Promise<SyncStatus>`
    - 通过SyncStatusRepository查询同步状态

  - `syncEventsPeriodically(programId: string, syncInterval: number): Promise<void>`
    - 使用setInterval定期调用syncEvents()
    - syncInterval: 同步间隔（毫秒），通过环境变量配置

| 用户状态 | User Stake Status | 单个用户在质押程序中的当前质押状态 |
| 用户活动 | User Activity | 用户在质押协议中的历史操作记录 |

| 程序 | Program | Solana上的智能合约 |

## 9. Business Rules (业务规则)

### 9.1 查询规则
- 全局状态查询：使用stakingMint和stateSeed（默认"state"）推导GlobalState的PDA，然后从链上实时查询账户
- 用户状态查询：先推导GlobalState的PDA，再使用stateAddress、userAddress和stakeSeed（默认"stake"）推导UserStakeInfo的PDA，然后从链上实时查询账户，并计算待领取奖励
- 历史记录查询：从数据库UserActivity表查询，按时间倒序

### 9.2 PDA Seeds配置规则
- stateSeed默认值为"state"
- stakeSeed默认值为"stake"
- 默认值来自合约常量定义，确保与链上合约一致

### 9.3 同步规则
- 事件必须按slot顺序同步
- 同步进度必须持久化，支持断点续传
- 同步失败需要记录日志，但不中断后续同步

## 10. Invariants (不变性约束)

### 10.1 事件同步不变性
- SyncStatus.lastSyncBlock >= SyncStatus.initializeBlock
- UserActivity中的事件必须按blockNumber递增
- 每个交易哈希在UserActivity中唯一

## 1. 查询功能实现 (Task 2)

### 1.1 查询GlobalState
**流程**:
1. 接收参数: programId, stakingMint, stateSeed (默认"state")
2. GlobalStateRepository.getGlobalState()
   - 推导PDA: `findProgramAddress([stateSeed, stakingMint], programId)`
   - 从Solana链查询GlobalState账户
3. 返回GlobalState实体

### 1.2 查询UserStakeStatus
**流程**:
1. 接收参数: userAddress, programId, stakingMint, stateSeed (默认"state"), stakeSeed (默认"stake")
2. UserStakePositionRepository.getUserStakePosition()
   - 先推导GlobalState的PDA
   - 再推导UserStakeInfo的PDA
   - 从Solana链查询UserStakeInfo账户
3. RewardCalculationService计算待领取奖励
4. 返回UserStakeStatus实体（包含pendingRewards）

## 2. 事件同步实现 (Task 3)

### 2.1 同步流程
**EventSyncService.syncEvents()**:
1. 获取SyncStatus，确定起始Slot
   - 如果SyncStatus不存在，从initializeBlock开始
   - 如果存在，从lastSyncBlock + 1开始

2. 调用EventSynchronizationService
   - 创建SolanaEventFetcher实例
   - 创建PermissionlessTransactionEventsParser实例
   - 调用SolanaEventFetcher.fetchEvents()获取链上事件

3. 将BaseEvent转换为UserActivity
   - PermissionlessStakedEvent → UserActivity (eventType: "Staked", positionDelta: amount)
   - PermissionlessUnstakedEvent → UserActivity (eventType: "Unstaked", positionDelta: -amount)
   - PermissionlessRewardsClaimedEvent → UserActivity (eventType: "RewardsClaimed", positionDelta: 0)

4. 保存UserActivity
   - 通过UserActivityRepository.save()批量保存
   - 检查txHash唯一性，避免重复

5. 更新SyncStatus
   - 更新lastSyncBlock为已同步的最大Slot

### 2.2 历史查询实现
**StakingQueryService.getUserActivityHistory()**:
1. 接收参数: userAddress, programId
2. UserActivityRepository.findByUser(userAddress, programId)
3. 按timestamp倒序排序
4. 返回UserActivity[]数组

## 3. 定时同步实现 (Task 4)

### 3.1 定时同步流程
**EventSyncService.syncEventsPeriodically()**:
1. 从环境变量读取SYNC_INTERVAL（默认60000毫秒，即60秒）
2. 使用setInterval定期调用syncEvents()
3. 错误处理：记录日志，不中断定时任务

### 3.2 启动流程
1. 应用启动时，为每个配置的programId创建定时任务
2. 定时任务按SYNC_INTERVAL间隔执行
3. 每次执行调用syncEvents()同步事件

## 4. 首次同步流程

### 4.1 首次同步步骤
```
1. 读取SyncStatus
   - 从环境变量读取PROGRAM_ID
   - 从SyncStatus表查询对应的program_id记录
   - 如果记录不存在，启动失败并退出
   - 读取last_sync_block和initialize_block

2. 确定同步范围
   - 如果last_sync_block为0，startSlot = initialize_block
   - 如果last_sync_block > 0，startSlot = last_sync_block + 1
   - endSlot: 当前Slot - slotToExclusion（由SolanaEventFetcher计算）

3. 执行同步
   - 调用EventSyncService.syncEvents(programId, startSlot, endSlot)
   - 从链上获取事件
   - 转换为UserActivity并保存
   - 更新SyncStatus.last_sync_block

4. 验证同步结果
   - 检查是否有事件被保存到UserActivity表
   - 验证SyncStatus.last_sync_block已更新
   - 记录同步统计信息（事件数量、耗时等）
```

## 5. 定时同步流程

### 5.1 定时同步执行
```
每次定时任务执行（间隔SYNC_INTERVAL）:

1. 读取SyncStatus
   - 获取当前program_id的last_sync_block

2. 确定同步范围
   - startSlot: last_sync_block + 1
   - endSlot: 当前Slot - slotToExclusion

3. 执行同步
   - 调用EventSyncService.syncEvents()
   - 如果startSlot > endSlot，跳过本次同步（已是最新）

4. 更新SyncStatus
   - 更新last_sync_block为已同步的最大Slot

5. 错误处理
   - 如果同步失败，记录错误日志
   - 不中断定时任务，下次继续执行
   - 如果连续失败，记录告警
```
## 6. 部署检查清单

### 6.1 部署前检查
- [ ] 数据库已创建并配置正确
- [ ] SyncStatus表已初始化，包含program_id记录
- [ ] initialize_block已正确设置（可通过查询InitializedEvent获得）
- [ ] 环境变量已配置（SOLANA_RPC_URL, PROGRAM_ID, STAKING_MINT等）

### 6.2 部署验证
- [ ] 服务启动成功，无致命错误
- [ ] 如果启动过程中没有致命报错，则认为部署成功

## 7. 故障处理

### 7.1 常见问题
1. **SyncStatus记录不存在**
   - 行为：启动失败，直接退出
   - 解决：执行数据库初始化步骤，插入SyncStatus记录后重新启动

2. **initialize_block为空或为0**
   - 行为：启动失败，直接退出
   - 解决：查询链上的InitializedEvent，获取正确的initialize_block值，更新SyncStatus记录后重新启动

3. **Solana RPC连接失败**
   - 行为：启动失败，直接退出
   - 解决：检查SOLANA_RPC_URL配置，确认RPC节点可访问

4. **同步无新事件**
   - 正常情况：如果链上确实没有新事件，同步会正常完成但无数据写入

5. **同步失败但定时任务继续**
   - 正常行为：错误已记录日志，下次定时任务会继续尝试

### 7.2 恢复流程
如果服务中断后需要恢复：
1. 检查SyncStatus.last_sync_block，确认同步进度
2. 重启服务，服务会自动从last_sync_block + 1继续同步
3. 验证同步是否正常恢复

Service Stop Workflow (服务停止流程)
================================================================================

## 停止步骤
```
1. 接收停止信号（SIGTERM或SIGINT）
   - 注册信号处理器

2. 停止定时同步任务
   - 清除所有setInterval定时器

3. 关闭数据库连接
   - 关闭数据库连接

4. 退出进程
   - process.exit(0)
```

================================================================================
Service Restart Workflow (服务重启流程)
================================================================================

## 重启步骤
```
1. 执行停止流程
   - 按照服务停止流程执行

2. 执行启动流程
   - 按照服务启动流程执行
   - 服务会从SyncStatus.last_sync_block + 1继续同步
```

## 重启说明
- 重启不会影响已同步的数据
- 服务会自动从last_sync_block + 1继续同步
- 不会重复同步已处理的事件（通过txHash唯一性保证）

## 模块详细说明

### 1. event-fetch (已存在)
- **类型**: Infrastructure Module
- **职责**: 链上事件获取和解析
- **独立性**: 高，不依赖其他业务模块
- **可复用性**: 高，可用于其他Solana项目

### 2. domain-models
- **类型**: Domain Module
- **职责**: 定义领域实体和值对象
- **独立性**: 高，纯领域模型，无外部依赖
- **可复用性**: 中，特定于staking业务

### 3. repositories
- **类型**: Infrastructure Module
- **职责**: 数据访问抽象
- **独立性**: 中，依赖domain-models
- **可复用性**: 中，特定于staking业务

### 4. domain-services
- **类型**: Domain Module
- **职责**: 核心业务逻辑
- **独立性**: 中，依赖domain-models和repositories
- **可复用性**: 中，特定于staking业务

### 5. query-service
- **类型**: Application Module
- **职责**: 查询功能的应用层
- **独立性**: 低，依赖多个模块
- **可复用性**: 低，特定应用

### 6. sync-service
- **类型**: Application Module
- **职责**: 同步功能的应用层
- **独立性**: 低，依赖多个模块
- **可复用性**: 低，特定应用
